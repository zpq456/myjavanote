#자바의 특징
자바는 자바 가상 머신을 써서 프로그램을 돌린다.
**JVM(자바가상머신)**은 자바 바이트코드를 실행할 수 있는 주체이다.
자바 바이트코드는 플랫폼에 독립적이며 모든 자바 가상 머신은
자바 가상 머신 규격에 정의된 대로 자바 바이트코드를 실행한다.
따라서 **C와는 다르게 모든 컴퓨터에서 실행이 가능하다.**

Java 컴파일을 하면 class파일을 생성하게 되고 JVM이 실행을
하게 된다. C와는 다르게 링크동작이 없다.

JDK는 자바 응용 개발 환경으로 개발을 하기 위해서 필요하고
JRE는 자바 실행 환경으로 실행을 하기 위해서 필요하다.
(JRE에는 JVM도 포함되어 있다.)

자바는 멀티스레드를 운영체제의 도움 없이 자체적으로 지원해준다.
멀티스레드는 프로그램당 1개씩만 할당되는 프로세스와는 다르게
다수의 스레드가 하나의 프로그램 안에서 병렬처리를 해준다.

##자바의 사용
자바의 기본타입
-boolean(1byte, true or false)
-char(2byte, Unicode)
-byte(1byte)
-short(2byte)
-int(4byte)
-long(8byte)
-float(4byte)
-double(8byte)

레퍼런스 타입
-클래스에 대한 레퍼런스
-인터페이스에 대한 레퍼런스
-배열에 대한 레퍼런스

상수의 사용
final를 붙여서 사용하며 초기값을 지정하고 실행중 값이
변경되지 않는다.

System.in
키보드와 연결된 자바의 표준 입력 스트림 으로 입력되는 키를
바이트로 리턴하는 저수준의 스트림이다. 이를 사용하면 바이트를 
문자나 숫자로 변환하는 많은 어려움이 있다.

class
클래스는 메모리에 할당되지 않고 형태만 있는것이며 이를 
사용하려면 객체를 생성하여 써야 한다. 비교하자면 클래스는
설명서 객체는 제품이라고 할 수 있다.

new
객체를 생성할때는 new를 사용하여 생성한다
(ex. a(변수명) = new b(클래스명))

자바의 예외처리
오류(Error)는 시스템적인 비상황이다. 예외(Exception)는 개발자가
구현한 로직에서 발생된다. 따라서 개발자가 예외처리를 해서
처리방법을 명확히 알고 적용해야 한다. 
자바는 예외(Exception)가 생겼을 때 해당 예외를 자신이 해결하거나 자신보다
상위의 클래스에 이를 넘겨 해결시키게 한다.
1. try~catch~finally
try구역에서 문제가 없으면 넘어가고 오류가 나면 해당 
오류일때의 catch구역을 진행시킨다. 이를 통해서 오류가
나면 그 클래스 안에서 해결을 할 수 있다.
finally는 try를 거치든 catch를 거치든 상관하지 않고
무조건 진행된다. 따라서 오류의 유무에 관계없이 꼭 해야
하는 작업을 넣어야 할 때 사용된다.
2. throwable 클래스
자신보다 한단계 상위의 클래스에 예외처리를 넘겨준다.
이때 throwable 클래스는 예외처리를 할 수 있는 최상위 클래스이다.

가비지 컬렉션
가비지 컬랙션은 가비지 컬렉터가 메모리의 힙 영역에 할당된
더 이상 사용되지 않는 객체를 다른 객체가 사용할 수 있도록
제거하는 것을 말한다.따라서 자바는 메모리 관리를 개발자가 
직접 해야 했던 C와는 다르게 자동으로 메모리 관리를 해 준다.

제네릭
클래스 내부에서 사용할 데이터 타입을 외부에서 지정하는 기법이다.
<>사이에 어떤 데이터 타입을 사용했느냐에 따라 달라지는데
데이터 타입을 확정하지 않고 인스턴스를 생성할 때 데이터 타입을
지정하는것이 가능하다.제네릭을 쓰게되면 코드중복을 막고 
타입 안정성을 높일 수 있다.

int와 Integer
자바에서 정수를 표현하는 자료형으로 int와 Integer를 사용한다.
int는 정수를 표현하는 자료형으로 우리가 C부터 사용해온 것이다.
그에 비해서 Integer는 클래스로 자바에서 사용하게 된 것이다.
이미 존재하는 int를 Integer로 만든것은 편의성을 제공하기 위해서
인데 Integer를 사용하면 null값 처리가 용이해서 sql과 연동할
경우 편리해진다.

###Git의 사용법
Git이란?
소프트웨어의 버전을 관리하기 위한 소프트웨어로 소스코드가
변경된 이력을 쉽게 확인할 수 있고 특정 시점에 저장된 버전과
비교도 가능하고 특정 시점으로 복원할 수 있다.
특징으로는 빠른속도에 단순한 구조 완벽한 분산과 비선형적인 
개발이 있다.

관리 시스템 종류
1. 중앙 집중식 버전 관리 시스템(CVCS : Centralized VCS)
서버 하나에 집중되는 형태로 관리하기가 무척 단순하지만
서버가 고장나서 데이터가 사라지면 코드 자체가 날라가
복수가 아주 힘들다.
2. 분산 버전 관리 시스템(DVCS : Distributed VCS)
여러군데에 분산하여 저장되고 개인마다 저장소가 있어
서버에 접속하지 않아도 개발이 가능하다.

Git의 작업 흐름
1. 작업환경에서 코드를 완성
2. 완성된 코드를 자신의 저장소에 커밋
3. 자신의 저장소에 커밋된 내용을 서버에 푸시
4. 푸시된 코드는 새 버전으로 저장됨.

Master와 Branch
Master는 현재 서버에 올려져 있는 완전하게 확인된 코드이며
Branch는 작업을 위해 해당되는 구역만을 개인 저장소로 가져오며
코드를 다 작성한 후에 서버에 올리면 검토 후 문제 없을 시에
Master의 코드와 합쳐준다.
